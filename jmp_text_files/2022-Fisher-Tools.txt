A likelihood function quantifies the probability of a set of data given a model with specific parameter values. Importantly, the likelihood function provides a bridge connecting a model to data, allowing the predictions of a model to be described in terms of probability distributions. It is difficult to underestimate the importance of the likelihood function, as it forms the basis for common statistical approaches to parameter estimation and model evaluation, including maximum likelihood estimation ( Myung, 2003 ), Bayesian parameter estimation and model comparison ( Gelman et al., 2013; Kruschke, 2014 ), minimum description length ( Grünwald, Myung, & Pitt, 2005 ), and various information criteria, such as Akaike information criterion ( Akaike, 1974 ), Bayesian information criterion ( Schwarz, 1978 ), leave-one-out cross-validation, and the widely applicable information criterion ( Vehtari, Gelman, & Gabry, 2017 ).

As we will discuss further below, some alternative approaches to parameter estimation commonly rely on minimizing the sum of squared error between the mean of data and the mean of simulated data. Unless the mean is a sufficient statistic, some information loss will occur, potentially biasing or distorting parameter estimates. The likelihood function circumvents this potential issue because it represents information about the entire distribution of data. This idea is supported by the likelihood principle, according to which, all information relevant for estimating parameters from data is contained in the likelihood function ( Berger & Wolpert, 1988; Birnbaum, 1962 ).

Despite its ubiquity and importance for parameter estimation and model evaluation, the likelihood function is not commonly used in a large class of models within the cognitive architecture approach to modeling. Cognitive architectures are computational instantiations of unified theories of cognition that attempt to explain how multiple cognitive processes function as a coherent system ( Newell, 1990 ). Defining a likelihood function for many models based on cognitive architectures is challenging due to their complex statistical structures, which often contain combinations of mixtures and convolutions. With few exceptions (i.e.  Weaver, 2008 ), the derivation of likelihood functions for cognitive architectures has been largely avoided. The lack of likelihood functions for cognitive architectures has hindered the adoption of the statistical model evaluation techniques described above. In addition to creating challenges for comparing a cognitive architecture to data, the lack of likelihood function complicates the comparison of cognitive architectures with each other, as well as to other modeling approaches.

Another potential barrier for adoption of likelihood based methods in the cognitive architecture community is the lack of tutorials tailored towards the unique challenges posed by cognitive architectures. Our goal in this tutorial is to survey fundamental tools that can be used to develop likelihood functions for cognitive architectures. The tutorial will focus on the cognitive architecture called Adaptive Control of Thought-Rational (ACT-R;  Anderson et al., 2004 ) because of its prevalence and empirical support across many domains, including memory ( Anderson, 1974; Anderson, Bothell, Lebiere, & Matessa, 1998 ), problem solving ( Anderson & Douglass, 2001 ), and multi-tasking ( Salvucci & Taatgen, 2008 ). Although likelihood functions enable the use of many evaluation methods, we will demonstrate the application of likelihood functions within the context of Bayesian parameter estimation because it has become a popular statistical framework within psychology during the past 10–15 years ( Winter, Ryan, Zondervan-Zwijnenburg, Depaoli, et al., 2017 ). The adoption of Bayesian methods has been spurred, in part, by criticisms of dominant frequentist methods ( Morey, Hoekstra, Rouder, Lee, & Wagenmakers, 2016; Wagenmakers, 2007 ), improvements in computing power and sampling algorithms ( Hoffman & Gelman, 2014; Smith & Roberts, 1993 ), and the development of a plethora of software packages for performing Bayesian analyses, including Jags ( Plummer et al., 2003 ), JASP ( JASP Team, 2019 ), PYMC3 ( Salvatier, Wiecki, & Fonnesbeck, 2016 ), Stan ( Carpenter et al., 2017 ), and Turing ( Ge, Xu, & Ghahramani, 2018 ).

The remainder of the tutorial will be structured as follows. We will begin with an overview of the ACT-R architecture and its core mechanisms. Then, we will describe limitations with current model evaluation techniques widely used in the cognitive architecture community and proceed to the next section where we will describe the likelihood function as an alternative. In the following section, we will describe a set of fundamental tools that can be used to characterize the statistical structure of ACT-R models and formulate likelihood functions to be used for Bayesian inference. We will then demonstrate how to apply some of these fundamental tools with examples taken from the standard ACT-R tutorial written in Lisp (see  Table 1 ). We selected models from the standard ACT-R tutorial for two reasons. First, most members of the ACT-R community are familiar with those models. Second, many of those models are at an appropriate level of complexity for a tutorial. The code used in this paper can be downloaded from the following repository:  https://osf.io/tjvm6/ . The repository includes Jupyter notebooks with annotated code for each model.

The ACT-R (Adaptive Control of Thought-Rational) cognitive architecture consists of a set of specialized information processing modules, responsible for a variety of functions, including goal directed behavior, vision, audition, motor activity, declarative memory and procedural memory among others ( Anderson, 2007; Anderson et al., 2004 ). As explained below, the procedural memory module coordinates the general flow of cognitive activity within the architecture by issuing requests to the other modules. In ACT-R, modules are subject to several information processing constraints that give rise to realistic behavior. Although some operations within a module might be massively parallel (such as memory retrieval), each module is limited to processing one request at a given time. Furthermore, the buffer for each module holds a maximum of one chunk of information, which is available to the procedural memory module. Collectively, these constraints on information processing produce errors and cognitive bottlenecks within the architecture.

The procedural memory module operates as a production system in which production rules encode and act upon procedural knowledge. Each production rule consists of two parts: (1) the conditions under which the rule is eligible for selection, and (2) a set of actions that are performed upon selection, such as executing a motor response. The process of selecting and executing a production rule is called a production cycle and a series of these cycles gives rise to cognition. During the selection process, which is called conflict resolution, a pattern matching mechanism in procedural module compares the conditions specified in production rules to the state of the modules. Using a utility calculation based in reinforcement learning, a single production rule is selected from among those with conditions that match the state of the buffers. Once a production rule is selected, its actions are performed, causing the state of the architecture to be modified. This production cycle is a serial process, but other modules can execute cognitive processes in parallel, depending on the structure of the model and the conditions and actions specified in the production rules.

The declarative memory system in ACT-R consists of a set of mechanisms for storing and retrieving memories of factual information. Much like the Generalized Context Model ( Nosofsky, 1986 ) and MINERVA-2 ( Hintzman, 1984 ), the basic unit of declarative memory in ACT-R is represented as a collection of feature-value pairs called a chunk ( Anderson et al., 1998 ). For example, the chunk {(name,Bob), (department,accounting)} could represent your co-worker Bob who works in accounting. When a retrieval request is submitted to the declarative memory module, the chunk with the highest activation is retrieved. Depending on the model, activation may consist of several components, such as similarity, spreading activation, learning and forgetting dynamics, and noise.

Cognitive architectures are instantiated as a discrete event simulation that completes the same task as a human subject using theoretically posited cognitive mechanisms and processes. The simulation of cognitive processes unfold in a detailed step-by-step fashion, as opposed to relying entirely on mathematical abstraction. On each run of the simulation, the model interacts with the task, and simulated responses and reaction times are recorded. If a model passes the initial hurdle of completing the task, the model’s predictions are then compared to human data. The model evaluation process typically involves comparing averaged simulated data to averaged empirical data in terms of root mean squared error (RMSE) or

Some model parameters are fixed at default values, whereas others are calibrated to empirical data. Currently, the cognitive architecture community has not converged on a standard approach for parameter estimation. As reviewed by  Kangasrääsiö et al. (2019) , the parameter estimation process may involve performing a grid search (e.g. brute force), iterative manual adjustment of parameters, or numeric search algorithms, such as the simplex method ( Nelder & Mead, 1965 ). In some cases, no parameter estimation method is specified.

Unfortunately, the model evaluation methods currently in use have several drawbacks and limitations. One major drawback is the inability to quantify parametric uncertainty, which can have implications for the interpretation of parameters and predictions of future data. Interpreting point estimates is problematic because many parameters values with differing interpretations might be relatively plausible. Without taking into consideration parametric uncertainty, one could be mislead by interpreting the point estimate by itself. Moreover, the reliance on point estimates can have downstream consequences for predicting future data because the range of predictions can be artificially constrained without taking into consideration predictions at other parameter values. Another drawback is the potential loss of information from averaging over sources of variability, such as individuals, and/or trials. Even worse, averaging can lead to distortions of information in some cases because an average value is not guaranteed to be representative of a typical person ( Brown & Heathcote, 2003; Estes, 1956; Siegler, 1987 ).

Determining the appropriate likelihood function is often the most challenging step when applying likelihood-based methods to cognitive architectures because such models may give rise to complex statistical structures, often involving mixtures, convolutions, and dynamic components. In some cases, the likelihood function might be mathematically intractable. Broadly speaking, there are two approaches for applying a likelihood function: (1) the analytic approach in which the likelihood function is mathematically derived in closed-form, and (2) the approximate approach in which the likelihood function is approximated through extensive simulation of the data-generating process. As we will discuss in further detail below, there are trade-offs between these approaches, such as computational efficiency and mathematical complexity.

The analytic and approximate approaches entail a trade-off between analytic and computational tractability. The analytic approach requires an upfront investment of time and effort to derive the likelihood function. In some cases, the effort required might be significant and finding a solution may not be guaranteed. However, when likelihood functions can be derived, they are often faster because they require only a single evaluation per observation or less, whereas simulation-based approximations require often require 1000 to 10,000 evaluations. For example,  Fisher, Houpt, and Gunzelmann (2020)  found that the likelihood function for an ACT-R model of the fan effect ( Anderson, 1974 ) was several orders of magnitude faster to evaluate than the simulation-based approximation. Approximate likelihood methods are particularly useful in cases in which the likelihood function is intractable or difficult to derive. In many cases, developing a simulation of the data generation process is easier than deriving the analytic likelihood function, but may require waiting hours, days or even weeks for an analysis to complete. However, recent developments in GPU programming provide some ability improve simulation run time ( Lin, Heathcote, & Holmes, 2019 ). We believe that both methods are valuable tools, and deciding which approach to use will depend on the complexity of the model and the goals of the research.

Bayesian inference is a statistical framework for updating beliefs about model parameters (or models themselves) as new data are observed. In this approach, initial expectations and uncertainty in beliefs about parameter values are represented as a prior distribution. When additional data are taken into account, the prior distribution is updated through Bayes’ rule to produce a posterior distribution, which represents updated beliefs in light of new data. Bayesian inference has several advantages as a method of model evaluation compared to standard methods typically used for cognitive architectures like ACT-R. First, Bayesian inference provides a way to represent uncertainty in parameter estimates as probability distributions, which aids in the interpretation of parameter estimates ( Gelman et al., 2013; Kruschke, 2011 ). Second, there is no loss or distortion of information due to averaging artifacts because the likelihood function is evaluated at the level of individual data points. Third, Bayesian methods incorporate parametric uncertainty in the prediction of future data using posterior predictive distributions.

Bayesian inference is performed using either analytical or numerical methods. Analytic methods are applicable to only a small set of simple models, such as a binomial model ( Kruschke, 2014 ). Typically, in such cases, the prior distribution and likelihood function are conjugate—meaning, the prior and posterior distributions have the same distributional form, albeit with potentially different parameters. One advantage of conjugate models is that the posterior distribution can be interpreted in terms of the same parameters as the prior distribution.

In most real-world cases, a closed-form solution to the posterior distribution is not available, either because the integrals do not simplify, or the dimensionality of the parameter space is prohibitively large. The standard numerical approximation is Monte Carlo Markov Chain (MCMC) sampling. A detailed tutorial for MCMC sampling is beyond the scope of this article. However, we will provide a brief overview and refer interested readers to  Van Ravenzwaaij, Cassey, and Brown (2018)  for a more detailed tutorial. MCMC sampling is a method for sampling a complex probability distribution whose properties are unknown. The posterior distribution is sampled through a series of transitions in the parameter space. The Markov chain transitions to a new proposed position if it yields a posterior log likelihood that is greater than the corresponding value of the current location. If the posterior log likelihood is lower for the proposal, it will transition to the proposed location with a probability equal to the ratio of their posterior log likelihoods.

Various algorithms have been developed for performing MCMC sampling, which differ primarily in the proposal process and have different advantages and disadvantages. In this tutorial, we will use the No U-Turn Sampler (NUTS  Hoffman & Gelman, 2014 ) and Differential Evolution (DE  Ter Braak, 2006 ), as both apply to a wide range of models, and are robust to correlation in the posterior distribution. NUTS is a highly efficient sampler that can be used with continuous parameters that can be differentiated. DE is less efficient compared to NUTS, but can be applied to continuous parameters, even if they cannot be differentiated.

In this tutorial model, we will use the Semantic model from the standard ACT-R tutorial ( Software, 2020 ) to illustrate the concepts of serial processes and mixtures. In so doing, we will instantiate ACT-R as a discrete time Markov process. The semantic model describes how questions about category membership are answered, such as “Is a canary an bird?”, which we will abbreviate as canary-bird. As illustrated in  Fig. 1 , category relationships are represented as a semantic network in which nodes correspond to concepts and directed edges correspond to membership. To answer a question about category membership, the model checks whether two nodes are connected directly or indirectly in the semantic network. For example, the model answers the question bird-animal directly by traversing from the bird node to the animal node using a single memory retrieval. By contrast, the model uses a process called category chaining to answer the question canary-animal, which involves traversing from the canary node to the bird node and then from the bird node to the animal node using successive memory retrievals.

The Semantic model we develop is identical to the one described in the standard ACT-R tutorial with two exceptions: (1) we added production rules to encode the stimuli and (2) we enabled partial matching in the declarative memory system to produce a more realistic transition matrix capable of generating errors.

In this section, we introduce fast Fourier transform (FFT) and illustrate how it can be used to calculate the distribution of a sum of random variables. Sums of random variables arise in various aspects of the ACT-R framework, particularly in the overall duration of a sequence of processes, which typically consists of encoding, retrieval, and action execution processes. Outside of a limited number of cases, the sum of random variables does not have tractable analytic solution. In ACT-R, for example, memory retrieval time follows a lognormal distribution whereas conflict resolution and perceptual-motor processes follow a uniform distribution. Unfortunately, there is not a simple analytic solution for describing the likelihood function for the sum of lognormal and uniform random variables.

We propose FFT as a computationally tractable method for convolving arbitrary random variables. FFT involves transforming each random variable to the spectral domain via its characteristic function or a numeric approximation, computing the product of the transformed random variables, and performing the inverse transformation to obtain the density. One advantage of this approach is its generality—it can be applied to any distribution, which allows us to preserve the distributional forms used in ACT-R. In  Fig. 4 , we provide several examples of convolving various random variables using FFT. One disadvantage of FFT is that it tends to be slower than analytic solutions. In addition, the use of complex numbers and intensive calculations present computational challenges for derivative-based MCMC samplers, such as the NUTS algorithm ( Hoffman & Gelman, 2014 ). Fortunately, non-derivative based samplers, such as differential evolution MCMC ( Turner, Sederberg, Brown, & Steyvers, 2013 ), are viable alternatives.

We will use the Count model from the standard ACT-R tutorial to illustrate how to define the likelihood for the sum of serial, independent, random variables using FFT. The Count model counts from a start number to an end number through a series of memory retrievals. In the original implementation, the Count model does not encode stimuli or emit a response because the goal was to simply illustrate control flow within ACT-R’s production system. Furthermore, each cognitive process had a deterministic execution time. In order to develop a statistical model, we perturb each cognitive process with noise and add production rules for stimulus encoding and response processes to generate reaction times.

The count model begins with a set of production rules to encode the start and end number displayed on a screen. The first production rule sets the goal to count, finds the stimulus, and creates a chunk in the imaginal buffer to hold the start and end numbers. The next production rule shifts visual attention to the stimulus before encoding it to visual module. Next, a production rule retrieves the meaning of the number. 
                            1 
                         
                         
                            1 
                            Base-level activation is set to 10 for numbers so that their retrieval times are negligible. 
                          Finally, the value associated with the value slot of the retrieved chunk is transferred to the imaginal buffer where it is added to the start slot. A similar process is repeated for stimulus for the end number.

Once the stimuli are encoded into the chunk in the imaginal buffer, the model proceeds through a series of retrievals of “count-order” chunks to increment from the start number to the end number. Each “count-order” chunk contains a slot for the current number and the next number (e.g.  
                            
                               { 
                               
                                  ( 
                                  current 
                                  , 
                                  1 
                                  ) 
                               
                               , 
                               
                                  ( 
                                  next 
                                  , 
                                  2 
                                  ) 
                               
                               } 
                            
                         ). The counting process begins with the selection of a production rule that issues a retrieval request for a “count-order” chunk with a current slot value equal to the start number. Upon retrieving the “count-order” chunk, an increment count production rule is fired to increment the count. This production rule sets the current count to the value of the “next” slot of the retrieved chunk and issues new retrieval request using the same “next” value for the “current” slot. In essence, this creates a loop or chain in which the count is incremented and the subsequent chunk is retrieved until the end number is reached. When the end number is reached, a “stop” production rule issues a command to press the “j” key.

Memory retrieval in ACT-R is a minimum time process because the chunk with the highest activation is retrieved and retrieval time is an inverse function of activation. In the following example, we will show how the Siegler addition model from the standard ACT-R tutorial ( Software, 2020 ) can be cast as a LNR model. In the experimental task ( Siegler & Shrager, 1984 ), children solve simple addition problems, such as “2 + 2 = ?”. On each trial, each addend is presented auditorially in succession and the solution is provided verbally. The model that we develop based on the LRN is identical to the model presented in the standard tutorial with the following exceptions. First, the lognormal race model uses normally distributed noise for activation rather than a logistic approximation to the normal distribution ( Lebière et al., 1994 ). Second, the retrieval threshold is treated a random variable rather than a deterministic quantity, allowing retrieval failure times to be variable. Third, retrieval failures are mapped to a unique response.

On each trial, the Siegler addition model proceeds through a deterministic chain of production rules that build the problem representation in the imaginal buffer, retrieves the answer, and responds with the retrieved answer. For each addend, the model fires a production rule that listens to the auditory stimulus, retrieves the addend from declarative memory, and adds the number to a problem representation chunk in the imaginal buffer. The resulting chunk is used as a retrieval request in the next production rule to obtain the answer from declarative memory. When the next production rule fires, it “harvests” the answer, which entails assigning the answer to a sum slot in the problem representation chunk. Finally, the model fires a production rule to vocalize the answer.

In this example, we will demonstrate how to model learning in the declarative memory system with the paired associates model from the standard ACT-R tutorial ( Software, 2020 ). Unlike other models in this tutorial, responses are not independently and identically distributed due to the learning mechanisms. Before proceeding, it is worth noting that the model we develop is only applicable to a special case in which there is a one-to-one mapping between the retrieved chunk and the observed response. In such cases, one can determine which chunk was retrieved and reinforced. However, in cases involving a many-to-one mapping, such as guessing or partial matching, it is not possible to determine which chunk was retrieved. The resulting likelihood function is a dynamic mixture over retrieved chunks and reinforcement states which grows as trials progress. A general solution that accounts for such cases is challenging and beyond the scope of this tutorial.

Over the course of eight blocks, subjects learn paired associates consisting of a unique word paired with a unique number. In each block, the same 20 paired associates are presented in a randomized order. On each trial, the word component of a randomly selected paired associate is presented for five seconds. Subjects are instructed to press the number key associated with the word. In contrast to the task described in the standard ACT-R tutorial, subjects are asked to press a designated key in the event of a retrieval failure as opposed to omitting a response. This modification to the task can improve the estimation of the retrieval threshold. After a deadline of five seconds, the corresponding number is presented as feedback for a duration of five seconds.

On each trial, the model visually encodes the word associate and creates a chunk in the imaginal buffer representing the paired associate for the current trial. After visual encoding, the word is added to the word slot of the chunk in the imaginal buffer and the model attempts to retrieve a chunk with the word associate in the word slot. If the chunk is retrieved, the model responds with the corresponding number in the number slot of the retrieved chunk. The model responds with a designated key in the case of a retrieval failure. After the five second deadline, the number associate is presented as feedback and the model visually encodes and adds the number to the number slot of the chunk in the imaginal buffer. The chunk is then cleared from the imaginal buffer, at which time the number of uses for that chunk is incremented by one. In some small number of cases, the response is truncated because the deadline is exceeded. In such cases, the model omits a response and proceeds to process the feedback.

The likelihood function provides a statistical foundation for numerous parameter estimation and model comparison methods. The adoption of these methods for cognitive architectures has been hindered by challenges in specifying a likelihood function, and the lack of tutorials tailored to these specific issues. In this tutorial, we introduced fundamental statistical tools and methods for developing likelihood functions. These fundamental tools are highly composable, allowing them to be combined flexibly to form likelihood functions for a wide range of models. Even though we focused on ACT-R, the methods we covered are applicable to other cognitive architectures, such as Epic ( Kieras & Meyer, 1997 ).

Although developing likelihood functions for cognitive architectures can be challenging, in many cases the benefits justify the costs. In particular, likelihood functions avoid the potential for averaging artifacts, instead allowing the model to be evaluated with respect to the full distribution of predictions. In addition, once a likelihood function is derived, it is possible to achieve the benefits of Bayesian inference, such as quantifying uncertainty in parameter estimates with probability distributions. Finally, the evaluation time of exact likelihood functions tends to be faster than simulation-based approximations ( Fisher et al., 2020 ).

Developing analytic likelihood functions for cognitive architectures is not without its challenges. It requires an upfront investment in terms of time and effort. The investment may not be worthwhile during initial prototyping, but could be beneficial in later model development and testing stages. In such cases, a modeler may consider simulation-based approximations ( Turner & Sederberg, 2014 ) during early development and deriving a likelihood function in later development. Likelihood functions for certain statistical structures remain challenging to derive. For example, some models that use ACT-R’s base-level learning mechanism produce dynamic mixtures arising from many-to-one mappings between memory retrievals and responses. Without the development of approximations, such models become intractable due to the combinatorial explosion of mixtures. A possible solution may involve marginalizing over an approximate distribution of reinforcement states to simplify the likelihood function. While deriving likelihood functions for these types of models remains an activate area of research, simulation-based approximations are a valuable stop-gap solution.

Despite the challenges with some types of models, our tutorial demonstrates that a wide range of models based on cognitive architectures can be instantiated in a Bayesian framework using a small set of fundamental statistical tools and methods. We hope that this tutorial will serve as an entry point for the adoption of Bayesian methods for cognitive architectures and spur further development in this area.
