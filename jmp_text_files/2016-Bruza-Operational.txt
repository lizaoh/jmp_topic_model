Dzhafarov and Kujala (2015)  have introduced a contextual probability theory called Contextuality-by-Default (C-b-D) which is based on three principles:  
                         
                            1. 
                            (Indexation by conditions): A random variable is identified (indexed, tagged) by all conditions under which its realizations are recorded. 
                         
                         
                            2. 
                            (Unrelatedness): Two or more random variables recorded under mutually incompatible conditions are stochastically unrelated, i.e., they possess no joint distribution. 
                         
                         
                            3. 
                            (Coupling) A set of pairwise stochastically unrelated random variables can be probabilistically coupled, i.e., imposed a joint distribution on; the choice of a coupling is generally non-unique. 
                         
                       It is curious with respect to the first principle that a similar line of thinking emerged in the field of computer programming languages in the nineteen sixties, particularly with the advent of a radical new language called ALGOL-60. Before ALGOL-60, programming languages such as FORTRAN featured variables that are always accessible from any part of the program. These variables were termed “global” variables because of this property. As programming languages evolved, global variables were seen as a cause of errors, or “bugs”. In order to help counter programming errors like the one just described, so called block structured programming languages were developed. A block is sometimes referred to as a “lexical scope”, which is a syntactically delineated fragment of a program. The term “scope” refers to the set of program constructs such as variable definitions that are valid within that particular syntactic fragment of program code. As a consequence, the same variable name can be used in two different scopes but preserve unique “identities”, where the identity is defined by its particular functional use in a given scope.                          X

The intuition we will develop in this article is that classical probability theory is in some ways similar to obsolete programming languages like FORTRAN where all variables are global. We contend that as a consequence modellers are unwittingly predisposed to the invalid assumption that a variable is “the same” across different conditions, which contravenes the first principle of C-b-D. This can lead to potential “bugs” in the development of probabilistic models.

An order effect is said to occur when a decision made in a non-comparative context differs from the decision taken in a comparative context. The corresponding experiment comprises two conditions: In one condition the subject is first asked to rate whether a document summary is topically related to the query, followed by a rating of how well the subject understands the information in the summary. This second condition is referred to the “comparative” context for the decision on topicality because a decision on understandability is asked first and may influence the subsequent decision on topicality.

In contrast, the semantics of scoped programming languages ensure that multiple incarnations of variables are safely handled behind the scenes so the probabilistic modeller need not be concerned about them.

The aim of this article is to provide the syntax and semantics of probabilistic programs, or “P-programs”, where scopes are the central feature. The use of syntactic scopes has two consequences. Firstly, it allows random variables be safely overloaded in a way that accords with the first principle of C-b-D. Secondly, as P-programs may have multiple scopes, each scope returns a probability distribution as an interim result. As it is desirable that a P-program returns a single model, the question arises as how to combine these interim results into a single probability distribution. This question is related to contextuality in the following way.  Abramsky and Brandenburger (2011)  have shown that contextuality can be abstractly formalized by obstructions to the existence of global sections in sheaf theory. The advantage of the sheaf-theoretic approach is that it provides the foundations of a  general  semantics of contextuality. In addition,  Abramsky (2015)  has discovered that an instance of this general formulation of contextuality is surprisingly present in relational database theory where the problem is to determine whether a universal relation exists for a set of relations such that these relations can be recovered from the universal relation via projection. ( Vorob’ev, 1962  as discussed in  Barbosa, 2014  considered this problem long before relational databases became a field). This formulation of contextuality is very similar to the problem just posed regarding interim results. This article will attempt to further develop this connection with database theory by presenting the operational semantics of P-programs in the well known database language SQL.

With the preceding as background, the overall intention of this article is to provide the syntax and semantics of a scoped probability theory in the form of probabilistic programs, or “P-programs”. P-programs aim to make the task of a probabilistic modeller easier by providing a convenient language to express probabilistic models. More specifically, the P-programs are guided by the following design considerations:  
                         
                            • 
                            Adherence to the first principle of C-b-D, i.e., permit the safe overloading of variable name used in different conditions; 
                         
                         
                            • 
                            A P-program returns a single model; 
                         
                         
                            • 
                            P-program semantics are based on relational database theory to allow both an efficient and readable operational semantics.

The previous section attempted to show that probability theory is similar to the older programming languages such as FORTRAN because all the variables are globally accessible. The implicit assumption behind a variable being declared global is that it is one and the same across different conditions. The order effects example in the previous section showed that this assumption, though readily made, can be invalid. We therefore introduce the notion of a scope to counter such errors. A P-program consists of a single global scope, and an arbitrary number of scopes which may be nested to arbitrary depth. The goal of a P-program is to return a single joint probability distribution representing a model of the phenomenon being studied.

A scope corresponds to a condition, or context, related to the phenomenon being studied. Scopes are syntactically delineated by square brackets, and are uniquely labelled, except for the global scope which is not labelled. The P-program of  Fig. 2 
                          shows a global scope containing two nested scopes labelled  x  and  y . For example,  x  may represent non-comparative context and  y  a comparative context in the order effects experiment of the previous section. We will see that the labels of the scopes will be crucial for adherence to the first C-b-D principle. The last statement of a scope must be a  return  statement which returns a probability distribution, which is a model of the phenomenon examined in the condition corresponding to the scope.

Discrete random variables are declared as the first statements within a scope via the reserved word  var . This declaration statement specifies the values over which the random variables range. In the P-program depicted in  Fig. 2 , variable  V  is the only variable declared in the global scope, and is therefore the only variable that can be referenced everywhere in the program. The fact that variable  V  is declared global to scopes  x  and  y  expresses the modeller’s assumption that the marginal distribution of variable  V  will not vary across the conditions  x  and  y . Invariance of the distribution is a necessary but not sufficient condition of the invariance of the identity of the variables: Two different random variables may very well be identically distributed.

Let us now consider the nested scopes in more detail. Scope  x  begins by defining variables  T  and  U  as being local to the scope as well as defining the discrete values that these variables range over. After the variable declarations, the syntax  P(T,U,V)[exp1.csv]  defines a probability distribution together with as the name of a data file used to initialize the distribution. For example, the file “exp1.csv” would contain data collected in the experimental condition corresponding to scope  x . Note that the same variables  T  and  U  are used to define another distribution in scope  y . According to the first C-b-D principle, the identities of these variables are treated as distinct.

At this point we will start introducing the semantics corresponding to syntactic elements in P-programs in the style of denotational semantics. In denotational semantics, syntactic elements are delimited by

In this article, the denotations will be statements in the database language SQL because of its readability as well as its ability to express efficient operational semantics.

We begin by specifying the semantics of the P-program statement  P(T,U,V)[exp1.csv]  which defines a probability distribution labelled  P  in the scope labelled  x . In P-programs, a label like  P  need only be unique  within  the given scope, in this case  x . This frees the programmer to use this label in scopes other than  x .  Fig. 3 
                          specifies the semantics. The semantics is based on the CREATE TABLE statement, which creates a database structure in the form of a table. A table has a name, in this case Px, followed by a series of statements that define the columns of the table. In a P-program there may be multiple definitions of probability distributions with the same label across scopes, it is therefore necessary to uniquely identify a given distribution. For this purpose, the semantics uses subscripts based on the scope, in this case Px. As scope labels are unique in P-programs, this ensures that probability distributions are uniquely identifiable.

The definition of a probability distribution involves a label identifying the distribution, e.g.,   P , followed by a list of variables delimited by brackets. These variables must be declared in the scope where the probability distribution is defined, or global to it. Recall that variable  T  and  U  are declared local to scope  x , whereas variable  V  is a global variable. The CREATE TABLE statement involves a series of column definitions corresponding to each of these variables.

A column definition includes a data type, in this case INTEGER, which specifies the domain of values over which the corresponding probabilistic variable can range. Recall that a scope returns a single probability distribution. Syntactically, this is expressed via the  return  statement. The distribution is returned from a scope so that it can be used elsewhere in the P-program. However, as distributions are declared local to a scope it is necessary to have a mechanism to uniquely refer to the distribution that has been returned from the scope. As the label of a scope is unique in a P-program, this label can be used to identify the distribution returned by the scope. For example, scope  x  returns the distribution  P1  via the statement  return P1 .

The corresponding SQL semantics are expressed in  Fig. 12 
                         . These SQL semantics create a table called x. As scopes must be uniquely identified this ensures that p-tables returned by scopes in P-programs can be uniquely identified. After the table has been created, all other tables and views in the scope are dropped. This DROP statement mirrors the stack-based execution architectures of computer programming languages where local variables in a scope are pushed onto a run-time stack. When the scope is exited the local variables are popped off the stack and the only variable remaining on the stack is the return variable, which can be accessed by the program after the scope has been exited.

The creating and dropping of views and tables within a scope offers flexibility to the modeller. For example, in scope  x  there is a view created around  P1  and this is also the case in scope  y . Note how in both cases respective views have different definitions by comparing the SQL semantics depicted in  Figs. 10 and 11 . In this way, the programmer is free to use  P1  to refer to a distribution that is local to a given scope and reuse this label in another scope for a different purpose. Because of the scope structure and the associated dropping of locally defined p-tables and views when the scope is exited, the programmer does not have to worry that these different uses of  P1  will become confounded and produce errors.

The goal of the PJOIN statement is to combine the probability distributions returned from scopes  x  and  y  into a single probability distribution labelled  P  which is returned as the final result of the P-program. SQL is expressive enough to allow numerous possibilities to join distributions. In this article, two straightforward and commonly used means of joining tables in SQL will be adopted, namely the PJOIN operation between two p-tables is either (1) a probabilistic variant of the natural join operation, or (2) a Cartesian product of the two p-tables. The distinction between the two depends on whether the marginal distributions of variables with the same label in different scopes are identical.

When the marginal distribution of variable T is not the same across scopes  x  and  y , the Cartesian product of p-tables returned by these scopes is employed with the automatic renaming of the T into T1 and T2. These semantics show the structure of the resulting p-table differs from the Cartesian product as there is a single instantiation of the variable T via x.T as in this case the variable has a single identity across scopes  x  and  y . In order to form the natural join, the condition x.T=y.T in the WHERE clause specifies that rows of the respective p-tables x and y are “stitched together” where the values of T across the rows of the respective component tables are equal. Stitching the rows together in this way delivered the required set of rows to form the basis of the joint distribution across the three columns: x.T,y.U,x.V. Similarly, the condition x.T=inv.T allows these rows to be stitched together with the rows of the inverted table inv, which allows the probabilities to be computed via x.p * y.p * inv.p with the resulting column labelled p as required for a p-table.

There are a number of theoretical results in relational database theory which make “acyclic” hypergraphs significant to providing the semantics of joining p-tables.  Wong (1997)  detailed the relationship between a Markov network and a relational database model. The key idea behind this connection was the discovery of the equivalence between probabilistic conditional independence and a generalized form of multivalued dependency (GEMVD). Multivalued dependencies are a particular form of mathematical function which are used to define constraints on the data stored within the database. This connection allows a Markov distribution to be constructed by joining component distributions together.  Wong (1997)  described this relationship between Markov distributions and relational database theory as “generalized acyclic join dependency” (GAJD) which was formalized in an extended relational database model. GAJDs are in turn related to acyclic hypergraphs.  Wong (2001)  proved that a joint distribution factorized on a acyclic hypergraph is equivalent to a GAJD.

For example,  Gyssens and Paredaens (1984, p. 86)  state the following: “Cyclicity implies that one or more attributes of the database are ‘overloaded’ … The reason why a database attribute such as PRODUCT might become overloaded is that it has more than one function; it might mean a product that is being supplied and a product being purchased”. This is clearly very similar to the issue that the first principle of C-b-D addresses.  Gyssens and Paredaens (1984)  suggest that a way to deal with cyclicity is to “split the ‘overloaded’ attributes into two or more new attributes” in order to render the schema acyclic. This is clearly similar to the first principle of C-b-D which would view PRODUCT as one variable in two conditions, hence according to this principle, PRODUCT should not be a single attribute, but two attributes: PRODUCT_SUPPLIED and PRODUCT_PURCHASED, a unique attribute for each condition.

Gyssens and Paredaens (1984)  show that in some cases a cyclic hypergraph can be decomposed into two hypergraphs; one acyclic and the other cyclic. They prove that the cyclic sub-graph is not further decomposable. Such cyclic graphs are referred to as “supercycles”.

The cyclic schema depicted in  Fig. 22  is an example of a simple supercycle. Here the “overloaded” variables are denoted on the edges of the graph. In such cases, each overloaded variable should be renamed into two variables. A straightforward semantics forms the Cartesian product of the four p-tables where renaming automatically occurs with respect to the overloaded variables.

Recall that  Abramsky (2015)  has discovered that the problem posed in relational database theory whether a universal relation exists for a set of component relations such that these relations can be recovered from the universal relation via projection is but an instance of what has been termed “contextuality”. In this work, “possibility tables” are used as a basis for analysing contextuality. The schema of a possibility table corresponds to joint outcomes of two random variables, e.g., given two random variables A and B ranging over  
                            
                               { 
                               0 
                               , 
                               1 
                               } 
                            
                         , then there would be four columns in the possibility table corresponding to the joint outcomes (0,0), (1,0), (0,1), and (1,1). Each row of the possibility table specifies a Boolean distribution on the possible joint outcomes. In addition, each row of the table is associated with a “measurement context”. The set of measurement contexts comprise the possible combinations of measurements that can be made together yielding directly accessible empirical observations.  Abramsky (2015)  shows that a row in a possibility table formally corresponds to a local section in sheaf theory. Contextuality is present when local sections cannot be “glued” into a global section.

In addition,  Abramsky (2015)  establishes connections between relational databases and contextuality theory and suggests further connections could be made with probabilistic databases. The present work can be considered as a step in this direction. The p-tables presented in this article are probabilistic variants of tables commonly found in relational databases. As p-tables differ structurally from possibility tables, the formal connection to sheaf theory offered by the latter is lost. However, one aspect does carry across: The semantics of the PJOIN are designed to “glue” p-tables together, which is analogous to the glueing of local sections in the sheaf-theoretic definition of contextuality just presented. The question now arises what the PJOIN semantics say about contextuality. For a system of p-tables with acyclic schema, there are two cases: (1) When the marginals of the p-tables being glued  do not  agree, the system is contextual. (In this case there will be automatic renaming of variables) (2) When the marginals of the p-tables being glued  do  agree, the PJOIN semantics yields a joint distribution, which implies non-contextuality. (The latter case corresponds to the “Vorob’ev condition”  Abramsky, 2015 ). In other words, glueing the component p-tables together into a universal p-table succeeds such that each component table can be recovered by projection.

For a system of p-tables with cyclic schema, the semantics of the PJOIN cannot determine whether the system is contextual. A possible way forward is to find a minimal acyclic augmentation of the given cyclic join dependency ( Liu et al., 2011 ). It is an open question whether such methods from the field of databases can be informative with respect to contextuality to the same degree that linear programming methods can ( Abramsky & Brandenburger, 2011; Dzhafarov & Kujala, 2012 ).

The consequence of P-programs allowing random variables being local to scopes means they are not be visible to each other. This is very different from standard probability theory where all random variables are global and hence are all visible to each other and a joint distribution across them can be defined. The interesting question to be addressed by future work is how this lack of visibility is to be interpreted. The second principle of C-b-D defines the “unrelatedness” of variables and states “Two or more random variables recorded under mutually incompatible conditions are stochastically unrelated, i.e., they possess no joint distribution”. Assume a variable  X  declared in scope  x  and a variable  Y  declared in scope  y , and the scopes in the P-program do correspond to “mutually incompatible conditions”, then the syntactic invisibility of these variables would imply that these variables possess no joint distribution. The third principle of C-b-D states that they can be “coupled” by having a joint distribution imposed on them. There is a substantial theory being developed around couplings ( Dzhafarov, Kujala, & Larsson, 2015; Kujala, Dzhafarov, & Larsson, 2015 ). Different systems being modelled allow different sets and types of couplings, and these sets and types can be used to characterize the systems. For example, the theory of contextuality based on maximal couplings involving cycles which parallel the notion of cyclic schema presented in this article. However, it is not clear how the semantics presented in this paper relates to this theory of contextuality and to what degree these database driven semantics can express it.

Currently the nascent field of probabilistic programming is giving rise to a number of different languages ( Gordon, Henzinger, Nori, & Rajamani, 2014 ). As a consequence, there is a diversity in program syntax depending largely on the style of computer programming language that was used as a starting point. For example, some probabilistic programming languages such as “STAN” and “Church” are based on the functional programming paradigm. To our knowledge, none of the development of these languages has been informed by the modelling challenges relating to the first principle of C-b-D. When modelling more complex phenomena, potentially errant assumptions can be made about a variable being the same across different conditions. For example, consider  Fig. 24 
                       taken from  Johnson et al. (2013) .

This diagram depicts an object-oriented belief network (OOBN) probabilistic model which features three subnetworks. The three subnetworks were created in parallel: a biological, ecological and human factors network, which were then combined to create a complete representation which aims to model the viability of the free-ranging cheetah population. Three modelling teams specified the interfaces for the subnetworks, which dictate the flow of information between different subnetworks. An interface for a subnetwork consists of input and output nodes. Nodes of interest to more than one subnetwork were identified. Each of these nodes would be “owned” by a particular OOBN subnetwork that would create it as an output node to make it visible to the other subnetworks. Note how variables such as “cheetah removal” are assumed to be the same variable in the different conditions represented by the three subnetworks. According to the C-b-D principle, such variables should be distinct, that is, there should be three separate variables for “cheetah removal”, one for each of the three conditions.

This article has attempted to set out a probabilistic programming approach where the operational semantics adhere to the first principle of Contextuality-by-Default (C-b-D). The advantage is that modellers can use syntactic scopes which allows them to safely overload variables. In this way, modellers will be able to more conveniently express their probabilistic models because they don’t need to keep track of the labelling conventions on variables advocated by the first principle of C-b-D. In addition, the article has demonstrated how theory from the field of relational databases can lead to a straightforward operational semantics of probabilistic programs via the well known database language SQL. Particular attention was paid to the semantics of the PJOIN operation which allows interim probability distributions to be “glued” to form a single distribution. The semantics of the PJOIN presented in this can determine contextuality when the schema of the component distributions are acyclic, but cannot do so when the schema are cyclic.
